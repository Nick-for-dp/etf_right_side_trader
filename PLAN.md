## 当前状态

当前版本：`v2.2-market` 初版已完成，进入复盘验证阶段。

生产能力：

- 日线数据链路：Tushare 历史 OHLCV、BaoStock 日常 OHLCV、AKShare/EastMoney NAV。

- 市场热度链路：宽基指数历史/日常行情、`market_regime` 计算、advisor 买入类门控。
- 展示链路：Streamlit 市场总览、持仓、详情、盈亏分析、策略对比。
- 回测链路：`backtest-odds` 对比 v2.0、v2.1A 赔率门控、v2.2 市场热度门控。

### P2.3：加仓冷却机制

目标：避免系统连续多日重复提示加仓，降低短期过度交易风险。加仓建议应尊重用户真实建仓/加仓记录，而不是只根据技术 BUY 机械提示。

第一版规则：

| 条件 | 处理 |
|---|---|
| 无持仓且触发 BUY | 可提示建仓 |
| 有持仓且触发 BUY，近 5 天无建仓/加仓记录 | 可提示加仓 |
| 有持仓且触发 BUY，近 5 天有建仓/加仓记录 | 降级为继续持有 |
| SELL 或风控触发 | 不受加仓冷却限制 |

实现建议：

| 任务 | 验收 |
|---|---|
| 交易记录建模 | 已新增 `trade_records`，记录 code、action、trade_date、price、shares |
| 枚举收敛 | 已新增 `SignalType / TradeAction / AdviceAction / SignalSource` |
| Dashboard 写入记录 | 用户在持仓页确认建仓/加仓/减仓时，同步写入交易记录 |
| advisor 接入冷却 | `generate_advice` 根据最近一次真实建仓/加仓日期拦截频繁加仓 |
| 配置化冷却天数 | 默认 5 天，可在 `settings.yaml` 调整 |
| 单测覆盖 | 覆盖近 5 天拦截、超过 5 天放行、SELL 不受影响 |

当前实现说明：

- `trade_records.action` 使用 `TradeAction = BUY / ADD / REDUCE / SELL`。
- Dashboard 建仓/加仓会调用 `PositionService.add` 更新持仓均价、份额和 `positions.entry_date`，随后写入交易流水。
- `positions.entry_date` 当前会在加仓时更新为最近一次买入日期；后续如需区分“首次建仓日”和“最近买入日”，应以 `trade_records` 为准。
- 为兼容老持仓，若 `trade_records` 暂无记录，advisor 会临时回退使用 `positions.entry_date` 做加仓冷却判断。

已知风险：

| 风险 | 影响 | 后续处理 |
|---|---|---|
| Dashboard 先更新 `positions` 再写 `trade_records`，暂未放在同一事务 | 若交易流水写入失败，可能导致持仓和流水不一致 | 将“更新持仓 + 写交易记录”下沉到 service，并使用同一 DB session 事务提交 |

### P2.4：开盘后执行防守区间

目标：T-1 日线信号生成后，给次日实际操作增加防守条件，避免 A 股开盘后快速下跌、冲高或剧烈波动时仍机械执行买入/卖出建议。

定位：

- 该能力属于执行层防守，不替代策略信号，不进入 `strategy` 层。
- 第一版生成“若开市后价格达到某区间，则不建议再买入/卖出”的条件提示。
- 在没有实时盘中行情前，不自动改写当天建议，只提示用户执行时复核。

建议规则：

| 原建议 | 防守提示 |
|---|---|
| 建仓/加仓 | 若开市后价格较 T-1 收盘明显下跌，说明昨日趋势可能失效，不建议再买入 |
| 建仓/加仓 | 若开市后价格较 T-1 收盘明显上涨，说明短线追高风险增加，不建议再买入 |
| 卖出 | 若开市后价格急跌，提示避免恐慌性卖出，需人工复核 |
| 卖出 | 若开市后价格明显反弹，提示卖出必要性需重新评估 |

验收：

| 任务 | 验收 |
|---|---|
| 防守参数配置 | 买入下跌/上涨阈值、卖出下跌/上涨阈值可配置 |
| advisor 生成提示 | 对建仓、加仓、卖出建议生成对应执行区间 |
| 可复盘保存 | 建议保存防守区间和基准价，Dashboard 可展示 |
| 不影响回测主口径 | 无盘中数据时不把提示当作已触发拦截 |

## P3：v2.3 基本面辅助

目标：基本面分两层推进：硬基本面先做长期估值门控，舆情基本面先做风险提示和人工复核，不直接鼓励买入。

硬基本面：

| 任务 | 验收 |
|---|---|
| ETF -> 跟踪指数映射 | 覆盖当前配置 ETF，无法确认的标记 `UNKNOWN` |
| 指数 PE/PB 数据补齐 | 至少覆盖宽基和主要行业 ETF 的跟踪指数 |
| 估值分位计算 | 输出 `CHEAP / FAIR / EXPENSIVE / UNKNOWN` |
| advisor 负向门控 | `EXPENSIVE` 只拦截建仓/加仓，不主动生成 BUY |

舆情基本面：

| 任务 | 验收 |
|---|---|
| 舆情数据采集 | 按 ETF、行业、指数聚合新闻/政策/公告/事件 |
| 摘要和标签 | 生成 `POSITIVE / NEUTRAL / NEGATIVE / RISK / UNKNOWN` |
| Dashboard 展示 | ETF 详情页显示近期舆情摘要和重大风险 |
| 强风险复核 | `RISK` 时建仓建议降级为“观望，需人工复核” |

约束：

- 硬基本面是慢变量，适合做长期门控。
- 舆情是快变量，第一版只做风险提示，不用正面舆情主动放大买入。
- 舆情摘要必须保留来源和日期，避免不可追溯的 LLM 结论。

## P4：v2.4 行业资金流看板

目标：观察行业短期热度和资金持续性，但第一阶段不进入核心买入门控。

| 任务 | 验收 |
|---|---|
| 行业资金流采集 | 保存每日行业净流入、主力净流入、排名 |
| ETF 行业映射 | 每只行业/主题 ETF 能映射到一个主行业或主题篮子 |
| 资金流状态 | 输出 `INFLOW / NEUTRAL / OUTFLOW / UNKNOWN` |
| Dashboard 展示 | 展示近 1/3/5/10 日累计流入、行业排名和趋势 |
| 观察回测 | 比较技术 BUY 时行业流入/流出的后续表现 |

进入 advisor 的前置条件：

- 历史数据覆盖足够长。
- 资金流能稳定改善胜率或最大回撤。
- 对行业映射不明确的 ETF 不做硬门控。

## P5：多因子回测和参数优化

目标：每新增一个外部判断，都必须可单独开关、可回测、可解释，避免因子叠加后无法判断收益来源。

| 任务 | 验收 |
|---|---|
| 全量 ETF 验证 | 输出 26 只 ETF 的 `v2.1A / v2.2 / v2.3 / v2.4` 对比 |
| 因子消融实验 | 单独开启市场、估值、舆情、资金流，比较边际贡献 |
| 参数网格 | 测试市场热度阈值、估值分位阈值、赔率阈值、风控参数 |
| 交易成本参数化 | 将回测成本从代码常量迁移到配置或 CLI 参数 |
| 风控优先级测试 | 单测覆盖止损、回撤止盈、技术 SELL、各类门控优先级 |

关注指标：

- 追高建仓次数是否下降。
- 假反弹买入是否减少。
- 最大回撤是否改善。
- 盈利交易是否被过度拦截。
- 短历史或数据缺失时是否正确回退到现有策略。

## 暂缓事项

- 自动下单：当前项目只做建议，不接券商交易接口。
- 高频或分钟级策略：当前数据模型和策略均为日线。
- LLM 直接生成交易信号：可以做解释和摘要，不进入核心决策链。
